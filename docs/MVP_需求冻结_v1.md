# MVP 需求冻结文档 v1

## 1. 项目目标

本项目是一个给家庭旅游组织者使用的动态记账与结算工具。

它只解决以下核心问题：

1. 每一笔消费到底是谁一起出的
2. 这笔消费应该按家均分还是按人数分摊
3. 每家先交了多少钱、又垫付了多少钱
4. 最后每家应该补多少钱或退多少钱

本项目不是旅游管理平台，不做社交，不做支付，不做复杂财务系统。

## 2. 目标用户

- 第一用户：旅游组织者，例如“团长”“发起人”“记账人”
- 使用设备：手机优先，平板次之
- 使用特征：不喜欢复杂表单，不熟悉技术术语，更接近纸笔记账习惯

## 3. MVP 必做范围

### 3.1 旅行账本

- 新建一次旅行
- 查看旅行列表
- 打开某次旅行继续记账

### 3.2 参与家庭

- 添加参与家庭
- 编辑家庭名称
- 设置每家默认人数
- 设置备注

默认按“家”作为记账主体，例如：张家、李家、王阿姨。

### 3.3 先交的钱

- 记录某家先交的钱
- 可记录多笔
- 单独展示总额
- 单独展示团费余额

界面术语统一使用“先交的钱”，不使用“预交金额”。

### 3.4 消费记录

每笔消费必须包含：

- 日期
- 类型
- 金额
- 谁付款
- 这笔谁一起出
- 分摊方式
- 备注

### 3.5 分摊方式

MVP 只支持两种：

1. 按家均分
2. 按人数分摊

不做自定义金额，不做权重分摊，不做混合分摊。

### 3.6 结算结果

最终必须输出每家：

- 应承担总额
- 先交的钱
- 实际垫付金额
- 已支付总额
- 最后应补 / 应退
- 总账和所有人的余额
- 查询某一家/某个人的历史记录

## 4. 暂不做范围

以下内容全部排除在 MVP 外：

- 登录注册
- 云同步
- 多管理员协作
- 权限系统
- 小票图片
- 发票识别
- 微信支付直连
- 行程管理
- 地图
- 聊天
- 导出复杂报表
- 最少转账路径优化
- 原生 App

## 5. 核心术语冻结

为了适合中老年用户，统一使用以下术语：

- 参与家庭：`参加的几家`
- 预交金额：`先交的钱`
- 消费记录：`记一笔`
- 参与分摊：`这笔谁一起出`
- 分摊方式：`怎么分`
- 结算汇总：`最后谁补谁退`
- 付款方：`谁付的钱`
- 团费余额：`先交的钱还剩多少`

## 6. 页面结构冻结

MVP 页面收敛为 4 个主界面。

### 6.1 旅行列表页

目标：进入某次旅行。

显示内容：

- 旅行名称
- 日期
- 参与几家
- 总花费
- 是否已结算

操作：

- 新建旅行
- 打开旅行

### 6.2 旅行详情页

目标：看全局并进入主流程。

显示内容：

- 总花费
- 先交的钱总额
- 团费余额
- 参与家庭数
- 最近几笔记录

主按钮只保留：

- `参加的几家`
- `先交的钱`
- `记一笔`
- `总账和余额`
- `查一家的明细`

### 6.3 记一笔页

目标：尽快录完一笔消费。

字段：

- 日期
- 类型
- 金额
- 谁付的钱
- 这笔谁一起出
- 怎么分（按家 / 按人数）
- 备注

交互规则：

- 默认日期为今天
- 默认带出上一笔的参与家庭
- 参与家庭使用勾选，不使用复杂下拉
- 金额优先数字键盘

### 6.4 总账和余额页

目标：随时查看总账和所有人的余额。

每家仅展示：

- 应承担
- 先交的钱
- 实际垫付
- 最后应补 / 应退
- 总账和所有人的余额
- 查询某一家/某个人的历史记录

不展示复杂算法，不展示技术字段，不展示过长流水。

### 6.5 查一家的明细页

目标：点开某一家后，直接看到余额和历史记录。

显示内容：

- 当前余额
- 应承担
- 先交的钱
- 实际垫付
- 历史记录（先交、垫付、分摊）

## 7. 数据模型冻结

所有金额统一使用“分”为单位存储。

### 7.1 Trip

- `id`
- `name`
- `startDate`
- `endDate`
- `note`
- `createdAt`
- `updatedAt`

### 7.2 Party

- `id`
- `tripId`
- `name`
- `defaultHeadcount`
- `note`
- `sortOrder`
- `active`

### 7.3 Deposit

- `id`
- `tripId`
- `partyId`
- `amountCents`
- `paidAt`
- `note`

说明：`Deposit` 是“先交的钱”，不是消费。

### 7.4 Expense

- `id`
- `tripId`
- `paidAt`
- `category`
- `title`
- `amountCents`
- `payerKind`（`party` 或 `pool`）
- `payerPartyId`
- `shareMode`（`by_party` 或 `by_headcount`）
- `note`

说明：

- `payerKind=party` 表示某家垫付
- `payerKind=pool` 表示使用“先交的钱余额”支付

### 7.5 ExpenseParticipant

- `id`
- `expenseId`
- `partyId`
- `headcountSnapshot`
- `weightSnapshot`
- `shareAmountCents`

说明：

- `headcountSnapshot` 必须快照保存
- 不允许未来修改家庭人数时改动旧账结果

### 7.6 SettlementSummary

该结构可以由计算得出，不强制持久化。

- `partyId`
- `totalShareCents`
- `depositCents`
- `directPaidCents`
- `totalPaidCents`
- `netCents`
- `status`

## 8. 记账规则冻结

### 8.1 先交的钱

- 属于该家的已付款
- 不参与单笔消费分摊
- 参与最终净额结算
- 同时进入团费余额

### 8.2 消费

每笔消费都必须回答 3 个问题：

1. 谁付的钱
2. 这笔谁一起出
3. 怎么分

### 8.3 按家均分

- 只在本笔被选中的家庭之间平均分摊

### 8.4 按人数分摊

- 只在本笔被选中的家庭之间按人数分摊
- 使用本笔 `headcountSnapshot`

### 8.5 取整规则

- 金额按分计算
- 分摊时先按比例向下取整
- 剩余的几分按最大余数法补齐
- 必须保证所有 `shareAmountCents` 之和等于原金额

## 9. 结算规则冻结

### 9.1 应承担总额

`应承担总额 = 该家所有消费分摊金额之和`

### 9.2 已支付总额

`已支付总额 = 先交的钱 + 实际垫付金额`

其中：

- `先交的钱 = 该家所有 Deposit 之和`
- `实际垫付金额 = payerKind=party 且 payerPartyId=该家 的 Expense 金额之和`

### 9.3 净额

`净额 = 已支付总额 - 应承担总额`

解释：

- `净额 > 0`：该家多付了，应该退回或应收
- `净额 < 0`：该家付少了，还需要补
- `净额 = 0`：该家刚好结清

### 9.4 团费余额

`团费余额 = 所有 Deposit 之和 - 所有 payerKind=pool 的 Expense 金额之和`

团费余额的作用是解释“先交的钱还剩多少”，不是另一套结算逻辑。

## 10. 交互原则冻结

- 全站大按钮、大字号、少表单
- 尽量使用生活化词语，不使用技术词
- 单页只做一件事
- 新增消费时优先复用上一笔的参与家庭
- 常用类型用快捷按钮，不要求手动输入分类
- 家庭选择优先勾选，不优先下拉框
- 最终页先展示结论，再允许查看明细

## 11. 技术路线冻结

MVP 技术路线固定为：

- `React`
- `TypeScript`
- `Vite`
- `IndexedDB`
- `PWA（轻量安装能力）`

当前不引入：

- 后端服务
- 账号系统
- 云数据库
- 实时协作

## 12. 验收标准冻结

MVP 至少通过以下验收：

1. 能创建一次旅行并录入 3 家以上参与家庭
2. 能记录多笔“先交的钱”
3. 能记录按家均分的消费
4. 能记录按人数分摊的消费
5. 能处理某家只参与其中一笔消费
6. 能处理使用“团费余额”支付的消费
7. 最终每家的应补 / 应退结果与手工验算一致
8. 金额取整后总账严格平衡

## 13. 变更控制

从本文件开始，以下内容视为冻结项：

- 术语
- 数据模型主结构
- 核心记账规则
- 核心结算规则
- MVP 页面边界

如要新增功能，必须先回答：

1. 是否直接提升“录入快、算得清、看得懂”
2. 是否会拖慢 7 天内交付
3. 是否会让页面和术语复杂化

若答案不明确，则一律延期。
