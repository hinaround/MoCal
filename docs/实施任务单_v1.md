# 实施任务单 v1

本文件用于把架构决策直接转成开发推进清单。

## 1. 阶段安排

### Phase 0：需求冻结

目标：统一术语、规则、页面、数据模型。

交付物：

- `docs/MVP_需求冻结_v1.md`
- `docs/验算样例_v1.md`
- `docs/实施任务单_v1.md`

验收标准：

- 所有参与开发的 AI 只引用这 3 份文件
- 不再新增未冻结字段和功能

是否必须冻结后再继续：必须。

### Phase 1：账本内核

目标：把记账和结算逻辑做成稳定纯函数。

交付物：

- `src/domain/types.ts`
- `src/domain/share.ts`
- `src/domain/settlement.ts`
- `src/domain/__tests__/...`

验收标准：

- 样例 1、2、3 全部跑通
- 总分摊金额始终等于原消费金额
- 修改家庭人数不会改动旧账结果

是否必须冻结后再继续：建议完成后再进 UI。

### Phase 2：极简 UI

目标：打通建账本、录家庭、记一笔、看结算。

交付物：

- `src/pages/TripList`
- `src/pages/TripDetail`
- `src/pages/ExpenseForm`
- `src/pages/Settlement`
- `src/app/PartyLedgerPanel.tsx`

验收标准：

- 一次旅行可以完整录入并得到结果
- 能随时查看总账和所有人的余额
- 点某一家后能看到这家的历史记录
- 手机上单手可完成主要操作

是否必须冻结后再继续：主流程稳定后再进下一阶段。

### Phase 3：移动端优化

目标：让中老年用户顺手使用。

交付物：

- 大按钮和大字号
- 快捷勾选
- 复制上一笔参与家庭
- 金额数字键盘优化
- 可安装 PWA

验收标准：

- 首次使用者可在讲解 1 次后独立录账

是否必须冻结后再继续：建议在真实试用前完成。

### Phase 4：验收与边界测试

目标：排除真实场景中的账目错误。

交付物：

- 边界场景测试清单
- 修复记录

验收标准：

- 临时加入
- 只吃一顿
- 团费余额支付
- 按人数分摊
- 取整补差
- 零金额/空参与人防呆

## 2. 角色分工

### 给 Cursor

负责：

- 前端页面实现
- 录账交互
- 移动端布局
- 页面文案落地

不负责：

- 擅自修改账目公式
- 擅自增加超出 MVP 的功能

### 给 Antigravity

负责：

- 审计样例是否通过
- 检查边界条件
- 检查金额守恒
- 检查异常输入与错误提示

不负责：

- 主导 UI 结构
- 扩展产品范围

### 给你

负责：

- 拍板术语
- 提供真实旅游场景样例
- 试用并指出哪里卡手
- 决定哪些需求延期

## 3. 建议目录结构

建议从一开始就保持很轻的前端结构：

```text
src/
  app/
  pages/
  components/
  domain/
  storage/
  utils/
```

约束：

- 结算逻辑放 `domain/`
- 数据读写放 `storage/`
- 页面不直接写复杂计算
